# Plan: Multi-platform Hakka IME Template Repo

> Source of truth for the architecture. Originally authored as a session
> planning note and copied here for long-term reference.

## TL;DR
Create a new repo `rime-hakka` built around **Rime** as the single shared engine. One Rime schema package (YAML dictionary + schema) per dialect becomes the single source of truth; the same package is consumed unchanged by Weasel (Windows), Squirrel (macOS), fcitx5-rime (Linux/KDE/GNOME), Trime (Android), and Hamster (iOS). Launch with Huiyang (Hagfa Pinyim), structured so Meixian/Sixian/etc. drop in as sibling folders. Gboard submission is tracked as a separate long-lead workstream, since Gboard does not host third-party IMEs.

## Architecture at a glance
- **Core = Rime schema package** (plum-compatible). One folder per dialect under `schemas/`.
- **Dictionary build** = Python pipeline that ingests `chars.csv` (hkilang/TTS right-hand column = Hagfa Pinyim) + this project's flashcard CSV + phrase corpora, emits `<dialect>.dict.yaml`.
- **Distribution artifacts** (all generated by CI from the same source):
  - `hakka-huiyang-rime-vX.Y.Z.zip` — plum/Weasel/Squirrel/fcitx5/Trime/Hamster all accept this.
  - `hakka-huiyang.schema.yaml` + `.dict.yaml` raw files (for Hamster in-app import via URL).
  - Optional `.deb` / Homebrew tap / Scoop manifest that drop files into the right per-frontend user dir.
- **Gboard** is handled out-of-band via Google's language-support submission process (docs-only workstream, no code).

## Phases

### Phase 1 — Repo bootstrap (parallel with Phase 2)
1. Create the new GitHub repo `rime-hakka` and scaffold the directory tree above.
2. Add `LICENSE` (MIT for code) and `LICENSE-DATA` (CC-BY-SA 4.0 for dictionary data).
3. Add `.github/workflows/ci.yml` running: YAML lint, `build_dict.py`, `validate_schema.py`, pytest, package smoke test.
4. Seed `README.md` with per-OS install matrix (Windows/Weasel, macOS/Squirrel, Linux/fcitx5-rime, Android/Trime, iOS/Hamster) + Gboard note.

### Phase 2 — Huiyang dictionary pipeline *(parallel with Phase 1)*
5. Pull `chars.csv` from `hkilang/TTS` — keep only the right-hand "Hagfa Pinyim" column (tone digits 1–6). Write the license/attribution into `sources/README.md`.
6. Write `build/build_dict.py`:
   - Read `chars-huiyang.csv` → single-char entries.
   - Merge `flashcards-huiyang.csv` → multi-char phrase entries (higher weight).
   - Output `schemas/huiyang/hakka_huiyang.dict.yaml` with proper Rime header (`name`, `version`, `sort: by_weight`, `use_preset_vocabulary: false`).
   - Resolve polyphones by emitting one entry per reading.
7. Write `build/validate_schema.py` to catch: malformed codes (non-`[a-z0-9]`), tone-digit out of 1–6, duplicate entries, missing syllable in speller alphabet, and presence of the toneless `derive/([1-6])//` rule.

### Phase 3 — Huiyang schema definition *(depends on 6)*
8. Author `schemas/huiyang/hakka_huiyang.schema.yaml`:
   - `schema/schema_id: hakka_huiyang`, name `惠陽客家話 (Hagfa Pinyim)`.
   - `speller/alphabet: "abcdefghijklmnopqrstuvwxyz123456"`.
   - Algebra: `derive/([1-6])//` for toneless fallback, plus abbrev rules.
   - `translator/dictionary: hakka_huiyang`, `prism: hakka_huiyang`.
   - `punctuator` preset: default.
   - Reference syndict/hakka as a design template; do **not** copy their dict (different dialect).
9. Add `opencc/` tables only if T→S conversion is wanted; default off.

### Phase 4 — Packaging & installers *(depends on 8)*
10. `build/package.py` produces `dist/hakka-huiyang-rime-vX.Y.Z.zip` containing the `.schema.yaml` + `.dict.yaml` + `opencc/` at the zip root (Trime/Hamster/plum convention).
11. Add `recipe.yaml` at repo root so `bash rime-install <user>/rime-hakka` works via plum.
12. **Windows installer**: Inno Setup script that (a) detects Weasel user dir (`%APPDATA%\Rime`), (b) copies files, (c) invokes `WeaselDeployer.exe /deploy`. Fallback: document the drag-schema-onto-rime-install.bat flow from plum.
13. **macOS installer**: unsigned `.pkg` via `pkgbuild` that copies to `~/Library/Rime/` and runs `Squirrel --reload`. Also provide a Homebrew tap formula (planned).
14. **Linux packaging**: Arch `PKGBUILD` + Debian `debian/` dropping schema into `/usr/share/rime-data/` (picked up by fcitx5-rime on both KDE Plasma and GNOME).
15. **Android docs**: `installers/android/README.md` explains installing Trime from F-Droid/Play, then "Schema Management → Import" on the release zip.
16. **iOS docs**: `installers/ios/README.md` explains installing Hamster from App Store, using its built-in web server / Files app import on the zip.

### Phase 5 — Verification *(depends on 10)*
17. `tests/test_roundtrip_librime.py`: use `librime-python` to feed a corpus of Hagfa Pinyim strings (including flashcard phrases like `ngit5 teu2`, `tien1 ho2`) and assert the first candidate matches the expected Hanzi, with both toned AND toneless input.
18. Manual verification matrix in `tests/MANUAL.md` — one checklist item per frontend (Weasel, Squirrel, fcitx5-rime on KDE, fcitx5-rime on GNOME, Trime, Hamster) typing the same 5 test phrases.
19. Release workflow: on git tag, build zip, run smoke tests, attach to GitHub Release with SHA256s.

### Phase 6 — Gboard submission *(independent, can start anytime after Phase 3)*
20. `gboard-submission/README.md` documenting:
   - Gboard does **not** accept third-party IMEs; only Google-curated language packs.
   - Pathway: Google's Language Community / in-app "Suggest a new language".
   - Prepare the required linguistic artifacts: phoneme inventory, tone table, 1k-word frequency list, sample sentences — all derivable from `chars-hkilang.csv` and phrase corpora.
21. Track submission status in a GitHub Issue template; this workstream is documentation/advocacy, not code.

### Phase 7 — Template-ify for additional dialects *(depends on 17)*
22. Extract `schemas/_template/` from the working Huiyang folder with `{{DIALECT_SLUG}}` / `{{DIALECT_NAME}}` placeholders and a `scripts/new_dialect.py` scaffolder.
23. Document in `CONTRIBUTING.md` how to add e.g. Meixian: copy `_template`, point `build_dict.py` at a Meixian source CSV, set schema metadata, ship.

## Relevant files / upstream references
- `hkilang/TTS` → `src/res/chars.csv` — primary single-character reading source (right column = Hagfa Pinyim, ~10k chars including polyphones).
- `syndict/hakka` → `hakka.schema.yaml`, `hakka_meixian.schema.yaml` — reference design for `speller.algebra` and `translator` blocks; don't copy dictionaries (different dialect).
- `rime/plum` → `rime-install` + `recipe.yaml` format — our distribution spec.
- `rime/weasel`, `rime/squirrel`, `fcitx/fcitx5-rime`, `osfans/trime`, `imfuxiao/Hamster` — five frontends, all consume the same schema/dict YAMLs.
- Hakka-flashcards-site's `Hakka Vocabulary.csv` — seed corpus for common phrases (boosted weight).

## Decisions
- Rime is the single engine everywhere except iOS, where Hamster (librime-based) is the compatible path. No separate native codebases per platform.
- Gboard is documentation-only; do not try to ship a Gboard "plugin" (not a thing).
- Windows goes via Weasel schema-package — no custom TSF IME.
- Launch scope: Huiyang only; `_template/` and `new_dialect.py` make Meixian/Sixian/etc. additive later.
- Data licensing: dictionary as CC-BY-SA 4.0 (matches hkilang/TTS and syndict/hakka precedent); code as MIT.
- **Tone-input UX**: toneless input is the default; tone digits 1–6 are accepted as optional disambiguation. Implemented in `speller.algebra` via a `derive/([1-6])//` rule so every toned entry also matches its toneless form; `speller.alphabet` still includes digits so users *can* type tones.
